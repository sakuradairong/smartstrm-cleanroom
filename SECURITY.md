# 安全策略

## 支持范围

本项目仍处于 clean-room 开发阶段，尚未发布稳定版本。安全修复只保证进入最新的 `main`；历史提交和第三方派生镜像不受支持。发布稳定版本后，本节会改为明确的版本支持表。

## 私下报告漏洞

请使用 GitHub 仓库的 **Security → Report a vulnerability**（Private Vulnerability Reporting）提交报告。不要为未修复漏洞创建公开 Issue、Discussion 或 Pull Request。

报告应尽量包含：

- 受影响提交、版本和部署方式；
- 漏洞类型、影响和前置条件；
- 不含真实凭据或个人媒体信息的最小复现；
- 建议修复或缓解方式（如有）。

请勿在测试中访问他人的实例、账号或数据，不要实施拒绝服务、持久化、数据删除或社会工程。若无法使用 GitHub 私密报告渠道，请等待维护者公布安全联系方式，不要改为公开披露。

## 响应流程

维护者会尽力在 7 天内确认收到报告，并在完成复现后沟通影响、修复计划和协调披露时间。实际时限取决于项目志愿维护能力，不能视为服务等级承诺。

修复发布后会在 [CHANGELOG.md](CHANGELOG.md) 中提供不扩大攻击面的说明，并在适当时申请 CVE/GHSA。

## 部署者责任

- 不把管理端口直接暴露到互联网；在可信网络或受控反向代理后运行。
- 更换所有示例口令和 Token，配置、备份、历史保持 `0600`。
- 保持 `sync_delete=false`，直到在独立目录完成预览和恢复演练。
- 使用固定镜像摘要或自行构建，并及时更新 Go、Alpine、ffmpeg 和反向代理。
- 第三方存储、TMDB、OpenList 和媒体服务器的漏洞应同时报告给对应上游。
